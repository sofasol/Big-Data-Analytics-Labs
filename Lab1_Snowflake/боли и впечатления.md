# Дневник выполнения лабораторной работы №1

## 1. Настройка окружения

Запустила контейнер PostgreSQL через Docker Compose.

## 2. Импорт данных

Сначала попробовала загрузить CSV через DBeaver. Первый файл встал нормально, но при попытке добавить второй вылезла ошибка с дублированием id. Загружать 10 файлов по одному вручную — долго, поэтому от этой идеи отказалась.

Решила сделать автоматический импорт через csvsql. Установила csvkit и psycopg2-binary в виртуальное окружение, написала скрипт import_data.sh, который за один проход загружает все 10 файлов в таблицу raw_mock_data. Скрипт отработал, в таблице оказалось 10000 строк.

## 3. Анализ сырых данных

После импорта изучила таблицу `raw_mock_data`. В ней 50 колонок, каждая строка — одна продажа. В строке смешана информация о покупателе, продавце, товаре, магазине, поставщике и самой сделке.

Выполнила несколько SQL-запросов. Посмотрела первые 10 строк, чтобы увидеть примеры данных. Проверила, какие поля уникальные: email покупателя и email продавца не повторяются, а названия товаров встречаются много раз. Посчитала количество уникальных магазинов, поставщиков и категорий товаров. Категорий оказалось три (Cage, Food, Toy), типов питомцев — пять (Dogs, Cats, Birds, Fish, Reptiles). Также убедилась, что все ключевые поля заполнены.

## 4. Проектирование схемы

На основе анализа спроектировала нормализованную схему снежинка. Получилось 12 таблиц: одна фактовая (`fact_sales`) и одиннадцать таблиц измерений. Измерения разбила на логические сущности: страны, города, типы питомцев, бренды, категории, покупатели, продавцы, магазины, поставщики, товары и питомцы.

## 5. Создание и заполнение таблиц

Написала DDL-скрипт для создания таблиц в правильном порядке: сначала независимые (страны, типы питомцев, бренды, категории), потом зависимые (города, покупатели, продавцы, магазины, товары), в конце таблица фактов.

Написала DML-скрипт для заполнения таблиц из `mock_data`. Использовала `DISTINCT ON` — это конструкция PostgreSQL, которая оставляет только одну строку для каждой группы уникальных значений. Например, `DISTINCT ON (supplier_name)` гарантирует, что каждый поставщик попадёт в таблицу один раз, даже если в исходных данных он встречается много раз. Это нужно, чтобы избежать дублирования в измерениях.

В процессе пришлось несколько раз корректировать порядок вставки, потому что одни таблицы ссылаются на другие. После исправлений скрипты отработали без ошибок.

## 6. Проверка результата

После запуска контейнера проверила, что `fact_sales` содержит 10000 строк — столько же, сколько и исходная `mock_data`.

Также сравнила суммы: посчитала общую выручку в сырой таблице (`sale_total_price`) и в таблице фактов (`total_price`). Обе суммы совпали. Это значит, что все данные перенесены корректно, ни одна продажа не потерялась и не продублировалась.

Ещё проверила, что в таблицах измерений нет дубликатов. Сделала это с помощью группировки с `HAVING COUNT(*) > 1`. Например, для покупателей:

```sql
SELECT customer_email, COUNT(*) FROM dim_customer GROUP BY customer_email HAVING COUNT(*) > 1;
```
Запрос вернул пустой результат — значит, дубликатов нет. То же самое проверила для продавцов, магазинов и брендов.

Также проверила внешние ключи. Например, для связи `fact_sales` с покупателями выполнила такой запрос:
```sql
SELECT COUNT(*) FROM fact_sales f 
LEFT JOIN dim_customer c ON f.customer_id = c.customer_id 
WHERE c.customer_id IS NULL;
```

Для удобства создала представления: `view_schema_summary` показывает количество записей в каждой таблице, `view_revenue_by_country` — выручку по странам, `view_top_brands` — топ брендов по выручке.

Все проверки пройдены. Схема полностью нормализована, данные перенесены без потерь. Ура!
